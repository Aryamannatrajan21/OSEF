# OSEF Architecture Guardrails

This is the architectural constitution of OSEF. Every pull request and future engineering decision must be validated against these immutable guardrails.

## 1. Extension Sandbox
- **Never expose internal SDK modules** to plugins. Plugins must interface exclusively via `osef.sdk`.
- **Never bypass Extension Context.** Plugins must only interact with the host system and graphs via the provided `ExtensionContext`.

## 2. Graph Immutability
- **Never mutate the Engineering Knowledge Graph from plugins.** The EKG is an immutable source of truth generated by the Core Parsers. Plugins may query it, but not alter it. (Transformation Engine will have its own pipeline).

## 3. Policy Abstraction
- **Never bypass the Engineering Policy Engine.** Analyzers must never hardcode engineering rules. All rules must be written as EPE `Rule` objects that yield `Finding` objects.
- **Never execute rules imperatively without DAG resolution.** The EPE manages execution order safely.

## 4. Language Agnosticism
- **Never introduce language-specific logic into Core.** OSEF Core must remain entirely agnostic. Language specifics belong exclusively in Parser Plugins.

## 5. Core vs. Ecosystem
- **Core owns abstractions. Plugins own functionality.** Do not bloat the Core with specific formatters (e.g., Markdown generation) or niche linting rules. Write a plugin.

## 6. Execution vs Observation Laws
1. **No External Network Calls** during Repository Scanning or Parsing.
2. **Strict Encapsulation**: The Symbol Table and Intermediate Representation formats are internal.
3. **Immutable Knowledge Graph**: Never mutate the EKG from plugins.
4. **No Magic**: Explicit dependency injection over global state.
5. **Execution vs Observation**: Execution is capability-driven. Observation is event-driven. Events never own execution state. The Event Bus is never RPC.
6. **Provider Statelessness**: Providers must remain completely stateless.
7. **Responsibility Laws**: Extension Host owns discovery. Capability Registry owns resolution. Pipeline Engine owns orchestration. Plugins own implementation.

*If an implementation violates these guardrails, stop, explain why, and request an Architecture Decision Record (ADR) override before proceeding.*
