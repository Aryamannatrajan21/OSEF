# OSEF Domain Models Specification

## Overview
This document defines the comprehensive domain models for OSEF, establishing the canonical language, responsibilities, lifecycle, and relationships for the core architectural domains. All definitions align with the Engineering Ontology Specification (EOS) v1.0.

---

## 1. Engineering Knowledge Domain

### Description
The canonical representation of engineering expertise, containing principles, standards, architectures, patterns, and playbooks.

### Properties
- **Responsibilities:** Store immutable engineering intent, power Capabilities, serve as the source of truth for the Agent runtime.
- **Lifecycle:** Drafted → Validated → Active → Superseded / Deprecated.
- **State Transitions:** Updates create a new version; older versions are marked Superseded.
- **Public Interfaces:** `KnowledgeProvider` (Protocol).
- **Events:** `KnowledgeLoaded`, `KnowledgeCreated`, `KnowledgeSuperseded`, `KnowledgeValidated`.
- **Dependencies:** None (Base tier of the ontology).
- **Versioning:** Semantic versioning per item.
- **Ownership:** OSEF Core Team / Community Governance.
- **Extension Points:** Custom storage adapters (via StorageProvider), Custom knowledge categories.

---

## 2. Plugins Domain

### Description
Independently installable extensions that contribute Capabilities to the framework without modifying the Core.

### Properties
- **Responsibilities:** Register Commands, Templates, Knowledge, Validators, Generators, Reviewers, Agents.
- **Lifecycle:** Discovered → Registered → Initialized → Active → Unloaded.
- **State Transitions:** Can be dynamically enabled/disabled per Session.
- **Public Interfaces:** `PluginProvider` (Protocol).
- **Events:** `PluginRegistered`, `PluginInitialized`, `PluginUnloaded`, `PluginError`.
- **Dependencies:** OSEF SDK, Service Contracts.
- **Versioning:** Independent Semantic Versioning per plugin.
- **Ownership:** Plugin Authors (Third-Party or First-Party).
- **Extension Points:** Hooks (e.g., `BeforeProjectInit`, `AfterValidation`).

---

## 3. AI Agents Domain

### Description
Autonomous reasoning components that orchestrate Workflows, executing Tasks based on context and Engineering Knowledge.

### Properties
- **Responsibilities:** Planning, Reviewing, Generating, Validating, Explaining.
- **Lifecycle:** Instantiated → Assigned Workflow → Executing Tasks → Completed / Failed.
- **State Transitions:** Idle → Thinking → Executing → Waiting For Validation → Completed.
- **Public Interfaces:** `AgentProvider` (Protocol).
- **Events:** `AgentStarted`, `AgentReasoning`, `AgentTaskCompleted`, `AgentFailed`.
- **Dependencies:** KnowledgeProvider, PromptProvider, Event Bus.
- **Versioning:** Inherits from OSEF Core or Plugin version.
- **Ownership:** Core Team or Plugin Authors.
- **Extension Points:** Custom Agent registration, specialized Tool access.

---

## 4. Commands Domain

### Description
User-facing execution directives invoked via the CLI, representing discrete Capabilities.

### Properties
- **Responsibilities:** Map user input to Workflows and Services. Provide help context.
- **Lifecycle:** Invoked → Validated Options → Executing Workflow → Terminated.
- **State Transitions:** N/A (Stateless execution boundaries).
- **Public Interfaces:** CLI Engine (`Typer` application structure).
- **Events:** `CommandInvoked`, `CommandCompleted`, `CommandFailed`.
- **Dependencies:** Services, Plugins (if provided by a plugin).
- **Versioning:** Bound to Core CLI or Plugin version.
- **Ownership:** CLI Engine.
- **Extension Points:** `Command Registration Hook`.

---

## 5. Templates Domain

### Description
Reusable blueprints for generating Artifacts, Prompts, or Code, populated by runtime context.

### Properties
- **Responsibilities:** Standardize outputs, enforce structural constraints without hardcoding content.
- **Lifecycle:** Loaded → Compiled → Rendered → Discarded.
- **State Transitions:** Uncompiled → Compiled.
- **Public Interfaces:** `TemplateProvider` (Protocol).
- **Events:** `TemplateRegistered`, `TemplateRendered`.
- **Dependencies:** Configuration, Context.
- **Versioning:** Versioned alongside the Knowledge or Plugin that provides it.
- **Ownership:** Generators, Plugins, Core.
- **Extension Points:** Custom Jinja filters, Template Overrides.

---

## 6. Prompts Domain

### Description
Dynamically assembled instructions for an AI model, generated by combining Templates with Engineering Knowledge and Session Context.

### Properties
- **Responsibilities:** Guide the Agent's reasoning strictly according to the Constitution and EKK.
- **Lifecycle:** Requested → Context Gathered → Rendered → Sent to AI → Logged/Discarded.
- **State Transitions:** Template → Populated Prompt → Executed Prompt.
- **Public Interfaces:** `PromptProvider` (Protocol).
- **Events:** `PromptGenerated`, `PromptExecuted`.
- **Dependencies:** KnowledgeProvider, TemplateProvider, Session Context.
- **Versioning:** Transient (Not versioned directly; generated from versioned components).
- **Ownership:** Prompt Engine.
- **Extension Points:** Custom Prompt Registration, Context Injectors.

---

## 7. Projects Domain

### Description
The software initiative being managed by OSEF, encompassing the Repository, configuration, and execution state.

### Properties
- **Responsibilities:** Maintain the scope and boundary of the current execution.
- **Lifecycle:** Initialized → Active (Under Analysis/Generation) → Closed.
- **State Transitions:** Uninitialized → Initialized.
- **Public Interfaces:** `ProjectContext` (Dataclass).
- **Events:** `ProjectInitialized`, `ProjectAnalyzed`.
- **Dependencies:** Repository, Configuration.
- **Versioning:** N/A (Runtime state).
- **Ownership:** The end-user (Session).
- **Extension Points:** Project Metadata attach points.

---

## 8. Repositories Domain

### Description
The version-controlled codebase containing source, documentation, and history. OSEF analyzes but does not own the repository.

### Properties
- **Responsibilities:** Provide source code boundaries, file access, and historical context (Git).
- **Lifecycle:** Discovered → Analyzed → Mutated (by Agents) → Committed.
- **State Transitions:** Clean → Dirty (Uncommitted changes).
- **Public Interfaces:** `RepositoryProvider` (Protocol).
- **Events:** `RepositoryAnalyzed`, `RepositoryMutated`.
- **Dependencies:** OS File System, Git.
- **Versioning:** Git SHAs / Tags.
- **Ownership:** The end-user.
- **Extension Points:** Custom Version Control Adapters.

---

## 9. Configuration Domain

### Description
The immutable settings defining how OSEF operates within a given Session or Project.

### Properties
- **Responsibilities:** Provide typed configuration to all Services (Logging, Paths, Provider selection).
- **Lifecycle:** Loaded → Validated → Active → Frozen.
- **State Transitions:** Unloaded → Loaded/Frozen.
- **Public Interfaces:** `ConfigurationProvider` (Protocol).
- **Events:** `ConfigurationLoaded`.
- **Dependencies:** OS Env, Config Files (`pyproject.toml`, `osef.yaml`).
- **Versioning:** Config Schema versioning.
- **Ownership:** Core System.
- **Extension Points:** Plugin Configuration schemas.

---

## 10. Validation Domain

### Description
Logic and Services responsible for verifying architectural, security, testing, and standard compliance.

### Properties
- **Responsibilities:** Assert that generated code or proposed architecture complies with the EKK and Constitution.
- **Lifecycle:** Triggered → Rules Fetched → Asserted → Passed/Failed.
- **State Transitions:** Pending → Passed | Failed.
- **Public Interfaces:** `ValidationProvider` (Protocol).
- **Events:** `ValidationStarted`, `ValidationPassed`, `ValidationFailed`.
- **Dependencies:** KnowledgeProvider, Context.
- **Versioning:** Tied to Core or providing Plugin.
- **Ownership:** Validation Engine.
- **Extension Points:** Custom Validator Hooks.
